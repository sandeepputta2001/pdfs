Platform Overview

PulseAnalytics is an event-driven analytics platform that collects behavioral telemetry from mobile/web SDKs, processes it through a multi-stage pipeline, and delivers real-time dashboards and cohort analytics — designed from ground-up to scale horizontally to 100M events/sec.

💡
Design Philosophy: Every layer is independently scalable. The ingestion tier scales independently from processing, which scales independently from storage. No single component is a bottleneck — this is guaranteed through design, not heroics.
📱
SDK Layer
CLIENT-SIDE
Lightweight SDKs for Android, iOS, Web, React Native, Flutter. Events are batched locally, compressed with gzip/brotli, and flushed on schedule or threshold. Offline buffering with SQLite/IndexedDB.
  • Event batching (max 500 events or 30s)
  • Auto-retry with exponential backoff
  • Device fingerprinting + anonymous ID
  • Session tracking, foreground/background
⚡
Ingest Gateway
GOLANG · LAYER-7
Stateless Go microservices behind AWS ALB/NLB. Validate, authenticate (API key / JWT), decompress, and publish to Kafka. Designed for 500K req/s per pod with minimal CPU.
  • Rate limiting via Redis token bucket
  • Request deduplication via Bloom filter (bloomd)
  • Schema validation (JSON Schema / Protobuf)
  • Admission control during overload
🚀
Kafka Streaming Bus
EVENT BACKBONE
Multi-cluster Kafka as the durable event bus. Topics partitioned by app_id for locality. Separate topics for raw events, processed events, and system events.
  • 3000+ partitions across 30+ brokers
  • Tiered storage (S3-backed) for cost
  • Separate clusters per region
  • MirrorMaker2 for cross-region replication
⚙️
Stream Processors
GOLANG · FLINK
Go-based stream consumers + Apache Flink for stateful aggregations. Real-time enrichment, deduplication, sessionization, and funnel computation.
  • Go consumers for raw enrichment
  • Flink for windowed aggregations
  • Redis for real-time state (sessions)
  • Dead-letter queue for failures
🗄️
Polyglot Storage
MULTI-ENGINE
Different storage engines for different query patterns. ClickHouse for OLAP analytics. PostgreSQL for metadata. MongoDB for flexible event schemas. Redis for hot aggregates.
  • ClickHouse: analytical queries, funnels
  • PostgreSQL: app config, user profiles
  • MongoDB: raw events (schema-flexible)
  • Redis: real-time counters + sessions
📊
Analytics API
GOLANG · GRPC + REST
Query serving layer with result caching, query planning, and multi-tenant isolation. Supports funnels, retention, cohorts, flows, dashboards.
  • gRPC for internal microservices
  • REST/GraphQL for dashboard frontend
  • Redis query result cache (L2)
  • Async query execution for heavy reports

Ingestion Pipeline

The ingestion layer is the most critical path. It must handle 100M events/sec with sub-100ms acknowledgment latency, provide durability guarantees, and protect downstream systems from overload.

SDK Design

GO · PROTO
// Event payload (Protobuf — ~200 bytes avg per event) message EventBatch { string app_id = 1; // tenant identifier string device_id = 2; // anonymous device uuid string user_id = 3; // post-login identity string sdk_version = 4; int64 sent_at_ms = 5; repeated Event events = 6; // batch of up to 500 } message Event { string event_id = 1; // client-generated UUID v4 string event_name = 2; // "app_installed", "purchase", etc int64 event_time = 3; // epoch ms (client clock) map<string, Value> props = 4; // arbitrary properties DeviceContext device = 5; AppContext app = 6; }

Ingest Gateway — Go Service

GO
func handleIngest(w http.ResponseWriter, r *http.Request) { // 1. Auth — API key lookup from Redis cache (1µs hot path) apiKey := r.Header.Get("X-API-Key") app, err := authCache.Get(ctx, apiKey) if err != nil { writeError(w, 401); return } // 2. Rate limit — token bucket per app_id in Redis if !rateLimiter.Allow(app.ID) { writeError(w, 429); return } // 3. Decompress + parse (gzip or brotli) batch, err := parseBatch(r) // 4. Deduplicate via Bloom filter (bloomd) newEvents := bloomFilter.FilterNew(batch.Events) // 5. Publish to Kafka — partition by app_id for ordering for _, evt := range newEvents { key := []byte(app.ID + ":" + evt.DeviceId) kafkaProducer.ProduceAsync("raw-events", key, evt) } // 6. Ack immediately — don't wait for Kafka ack (acks=1) w.WriteHeader(202) // Accepted }
⚠️
Deduplication Strategy: Client-generated event_ids are checked against a Bloom filter (bloomd) with ~0.1% false positive rate. This handles network retries cheaply. For exact deduplication, a second-pass check in the stream processor uses Redis SET with 24h TTL per event_id before writing to ClickHouse.

Stream Processing

Two-tier processing: Go microservices for fast per-event transformations, Apache Flink for stateful windowed aggregations. All processors are Kafka consumer groups — independently scalable.

1
Enrichment Service (Go)
Reads from raw-events. Appends GeoIP lookup (MaxMind mmdb, cached in memory), User Agent parsing, app version normalization, server-side timestamp correction (client_offset = server_time - client_time), and timezone resolution. Outputs to enriched-events.
~50 pods · 2M events/sec/pod · Go concurrent consumers
2
Session Engine (Go + Redis)
Reads from enriched-events. Groups events into sessions using a 30-minute inactivity timeout. Session state stored in Redis with TTL. On session close: computes session_duration, event_count, entry/exit pages. Emits session_start and session_end synthetic events to session-events.
Redis HASH per device_id · 30m TTL sliding · partitioned by device
3
Flink Aggregation Jobs
Apache Flink handles time-windowed aggregations: tumbling 1-minute windows for real-time counters, sliding 1-hour windows for hourly metrics, daily windows for DAU/MAU. State stored in RocksDB (Flink state backend). Checkpoints every 30s to S3. Outputs pre-aggregated rows to ClickHouse and Redis counters.
Flink 1.18 · RocksDB backend · exactly-once · S3 checkpoints
4
Funnel Processor (Go)
Stateful processor that tracks funnel step completion per user within configurable time windows (e.g., "install → register → purchase within 7 days"). State persisted in Redis sorted sets (score = event_timestamp). On each step event, checks if previous steps exist within window. Writes funnel_conversion events when complete.
Redis ZSET per user per funnel · async state eviction
5
ClickHouse Writer (Go)
Batch consumer that accumulates events in memory and performs bulk inserts to ClickHouse every 1 second (100K-1M row batches). Uses async inserts with deduplication enabled. Partitioned by (toYYYYMMDD(event_time), app_id) for optimal query pruning.
Bulk insert · 1s flush · async_insert=1 · dedup_token = event_id

Flink Job — Retention Computation

JAVA · FLINK
// Day-N Retention: users who installed on day X and returned on day X+N public class RetentionJob extends RichFlatMapFunction<Event, RetentionResult> { private ValueState<Long> installTimestamp; // Flink state @Override public void flatMap(Event e, Collector<RetentionResult> out) { if (e.name.equals("app_installed")) { installTimestamp.update(e.eventTimeMs); return; } Long installTs = installTimestamp.value(); if (installTs == null) return; long dayN = daysBetween(installTs, e.eventTimeMs); if (dayN >= 1 && dayN <= 30) { out.collect(new RetentionResult(e.userId, installTs, dayN)); } } } // Keyed by user_id → each user's state is isolated // Output → ClickHouse retention_events table
Storage Architecture

Polyglot persistence: each store is chosen for its query pattern, not convenience. Hot data is Redis. OLAP is ClickHouse. Metadata is PostgreSQL. Schema-flexible raw events are MongoDB. Cold archive is S3/GCS in Parquet.

⚡
ClickHouse — OLAP Core
PRIMARY ANALYTICS ENGINE
Why: Columnar storage with vectorized execution. Compresses 10:1. Scans 10B rows/s per node. MergeTree engine with partition pruning by date + app_id eliminates irrelevant data before scan.
  • Sharded by consistent hash on app_id
  • 3 replicas per shard (ReplicatedMergeTree)
  • Distributed table for transparent routing
  • Materialized Views for pre-aggregation
  • TTL policies: hot (SSD) → warm (HDD) → cold (S3)
  • Query result cache (query_cache TTL=300s)
🐘
PostgreSQL — Metadata
AURORA POSTGRESQL · HA
Why: ACID transactions for app config, user identity resolution, funnel definitions, cohort definitions, dashboard configs, billing. Read replicas for reporting queries.
  • Aurora PostgreSQL Multi-AZ (HA)
  • Read replicas for analytics metadata
  • PgBouncer connection pooling
  • Logical replication to reporting replica
  • Sharded by org_id for large tenants
🍃
MongoDB — Raw Events
ATLAS · SHARDED CLUSTER
Why: Schema-flexible document store for raw event storage before ClickHouse write confirms. Also stores user profiles with flexible property maps. Allows arbitrary property lookups and event schema evolution.
  • Sharded by (app_id, date) — range shard key
  • TTL indexes: raw events expire in 90 days
  • Atlas Search for property-based event search
  • Change streams for real-time triggers
🔴
Redis — Hot Layer
CLUSTER MODE · 32+ NODES
Why: Sub-millisecond reads/writes for real-time counters, session state, rate limiting, auth cache, query result cache. Redis Cluster with 32 shards.
  • Real-time event counters (INCR)
  • Session state (HASH + TTL)
  • Rate limiting (token bucket)
  • Auth API key cache (5min TTL)
  • Query result cache (L1 hot cache)
  • Pub/Sub for real-time dashboard updates
🪣
S3/GCS — Cold Archive
OBJECT STORE · PARQUET
Why: Cheapest storage for historical data. ClickHouse tiered storage moves data to S3 after 30 days. Raw Kafka events archived to S3 via Kafka S3 Connector for compliance and replay.
  • Parquet format with Snappy compression
  • Partitioned by year/month/day/app_id
  • Athena/BigQuery for ad-hoc historical queries
  • Glacier Deep Archive after 1 year
  • Lifecycle policies for cost optimization
🎯
Qdrant — Vector Store
AI FEATURES · RAG
Why: Powers AI Insights feature — natural language queries over analytics. Dashboard configs, metric descriptions, and event schemas are embedded and indexed for semantic search.
  • OpenAI / Cohere embeddings
  • RAG pipeline for "Why did DAU drop?"
  • Anomaly context retrieval
  • Deployed on GKE with persistent volumes

ClickHouse Schema — Core Events Table

SQL · CLICKHOUSE
CREATE TABLE events ( app_id LowCardinality(String), event_id UUID DEFAULT generateUUIDv4(), user_id String, device_id String, event_name LowCardinality(String), event_time DateTime64(3, 'UTC'), session_id String, country_code LowCardinality(FixedString(2)), platform LowCardinality(String), -- android/ios/web app_version LowCardinality(String), props Map(String, String), -- flexible properties revenue Nullable(Float64), INDEX props_idx props TYPE bloom_filter(0.01) GRANULARITY 1 ) ENGINE = ReplicatedMergeTree PARTITION BY (toYYYYMMDD(event_time), app_id) ORDER BY (app_id, event_name, toStartOfHour(event_time), user_id) TTL event_time + INTERVAL 90 DAY TO DISK 's3_cold' SETTINGS index_granularity = 8192; -- Materialized view for DAU pre-aggregation CREATE MATERIALIZED VIEW dau_mv ENGINE = SummingMergeTree PARTITION BY toYYYYMM(event_date) ORDER BY (app_id, event_date, platform) AS SELECT app_id, toDate(event_time) AS event_date, platform, uniqState(user_id) AS dau_state FROM events GROUP BY app_id, event_date, platform;

Query Serving Layer

Multi-tier query serving with intelligent caching, async execution for heavy reports, and a dedicated AI-powered natural language query interface.

🔌
Query API Service
GO · GRPC + REST
Stateless Go service. Receives query requests, translates to ClickHouse SQL, checks cache, and routes to appropriate backend.
  • gRPC for internal microservices
  • REST/JSON for external API consumers
  • GraphQL for dashboard BFF layer
  • OpenAPI spec + SDK generation
⚡
Caching Strategy
3-TIER CACHE
L1: In-process Go cache (ristretto, 1GB). L2: Redis cluster (5min TTL, shared). L3: ClickHouse query_cache (server-side, 300s TTL). Cache key = SHA256(normalized_sql + tenant_id).
  • L1 hit rate target: 40%
  • L2 hit rate target: 70%
  • Cache warming on dashboard load
  • Invalidation on new data checkpoint
📊
Async Query Engine
HEAVY REPORTS
Heavy queries (cohort analysis, LTV projections, custom SQL) are executed asynchronously. Results stored in S3, URL returned. WebSocket notification when complete.
  • Job queue in PostgreSQL
  • Result stored to S3 as CSV/JSON
  • Pre-signed S3 URL for download
  • Max 5 concurrent heavy queries/tenant
🤖
AI Insights (RAG)
QDRANT · LLM
Natural language → ClickHouse SQL. User asks "Why did retention drop last week?" — system retrieves relevant metric context from Qdrant, constructs prompt, LLM generates SQL + explanation.
  • Event schema embedded in Qdrant
  • Few-shot SQL generation prompts
  • SQL validation before execution
  • Anomaly context auto-retrieved

Multi-Tenancy & Isolation

🏢
Tenant Isolation Model: app_id is always included in every ClickHouse query predicate (enforced at the query API layer — no escape). ClickHouse row-level policies restrict access. Each tenant gets a resource quota enforced by the Query API: max concurrent queries, max scan bytes/hour, max result rows.

Sample Analytics Query — Funnel

SQL · CLICKHOUSE
-- 3-step funnel: Install → Register → Purchase, within 7 days SELECT windowFunnel(7 * 86400)(toUnixTimestamp(event_time), event_name = 'app_installed', event_name = 'user_registered', event_name = 'purchase_completed' ) AS level, count() AS users FROM events WHERE app_id = 'your_app_id' AND event_time BETWEEN '2024-01-01' AND '2024-01-31' AND event_name IN ('app_installed', 'user_registered', 'purchase_completed') GROUP BY user_id ORDER BY level DESC; -- Runs in <200ms on 10B rows with proper partition pruning + PREWHERE

✅
Scaling to 100M events/s is achieved by horizontal scaling at every tier: Gateway pods scale via HPA → Kafka partitions are pre-allocated → Stream processor consumer groups scale with partition count → ClickHouse shards distribute write load → Redis cluster scales by adding nodes. No single component is a bottleneck because each tier consumes from the previous tier's durable queue (Kafka).
Infrastructure & CI/CD

Kubernetes-native on GKE/EKS. Infrastructure as code with Tanka + Jsonnet. GitOps with ArgoCD. Full observability with Prometheus + Grafana + OpenTelemetry.

☸️
Kubernetes (GKE)
CONTAINER ORCHESTRATION
GKE Autopilot for stateless services (gateway, processors). GKE Standard for stateful (Kafka, ClickHouse, Redis). Node auto-provisioning with workload-optimized machine families.
  • Separate node pools per workload type
  • Spot/Preemptible for batch processors
  • PodDisruptionBudgets for zero-downtime
  • KEDA for Kafka consumer autoscaling
  • Vertical Pod Autoscaler for right-sizing
🔧
Tanka + Jsonnet
INFRA AS CODE
All K8s manifests defined in Jsonnet using Tanka. Environments (dev/staging/prod) share a single config with environment-specific overrides. Grafana Tanka for deployment.
  • DRY config with Jsonnet functions
  • Environment overlays (dev/staging/prod)
  • Secret management via Sealed Secrets
  • Diff preview before apply
🚀
CI/CD Pipeline
GITHUB ACTIONS · ARGOCD
GitHub Actions for build/test/push. ArgoCD for GitOps deployment. Image built, scanned (Trivy), pushed to ECR, then GitOps repo updated to trigger ArgoCD sync.
  • PR → lint + test + build → Docker image
  • Trivy vulnerability scan (block on CRITICAL)
  • Push to ECR with immutable tags
  • ArgoCD auto-sync on main branch
  • Canary deployments via Argo Rollouts
  • Automated rollback on error rate spike
📈
Prometheus + Grafana
OBSERVABILITY
Prometheus federation with Thanos for long-term storage. Pre-built Grafana dashboards for every service. Alertmanager → PagerDuty for on-call.
  • Thanos for HA + long-term metrics
  • Custom dashboards per microservice
  • SLO tracking (error budget dashboards)
  • Kafka lag alerting per consumer group
  • ClickHouse query latency P50/P95/P99
🔭
OpenTelemetry
DISTRIBUTED TRACING
OTEL SDK in all Go services. OTEL Collector as sidecar/DaemonSet. Traces exported to Jaeger (hot) and S3 (cold). Trace sampling at 1% for normal, 100% for errors.
  • Auto-instrumentation for gRPC/HTTP
  • Custom spans for Kafka produce/consume
  • Trace correlation in logs (trace_id field)
  • Tail-based sampling for error traces
🔒
Security
ZERO-TRUST
mTLS between all internal services via Istio service mesh. Network policies isolate namespaces. Secrets via HashiCorp Vault. RBAC with least-privilege everywhere.
  • Istio service mesh with mTLS
  • HashiCorp Vault for secrets rotation
  • OPA/Gatekeeper for policy enforcement
  • Falco for runtime threat detection
  • WAF on ingress (AWS WAF / Cloudflare)
